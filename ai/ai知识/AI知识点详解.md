# 第一章：Python 异步编程深度理解

## 1.1 同步 vs 异步的本质区别

**同步（Synchronous）**：代码顺序执行，一个任务完成才开始下一个。当任务在等待 IO（网络请求、文件读取）时，CPU 空转等待，线程被阻塞。

**异步（Asynchronous）**：当一个任务在等待 IO 时，CPU 立即切换到另一个任务。不是"同时做多件事"，而是"一件事卡住时先做另一件"。

类比：同步 = 一个人排队买咖啡，前面的人点完站那等咖啡做好，后面所有人都得等。异步 = 点完咖啡拿个号牌去旁边等，咖啡师继续接下一个人。

## 1.2 事件循环（Event Loop）工作机制

事件循环是异步编程的调度中心——一个无限循环，不断从任务队列中取出协程并执行。

**执行过程**：
1. 注册协程到事件循环
2. 事件循环开始执行协程，遇到 `await` 时暂停当前协程
3. 暂停的协程注册到 IO 事件监听
4. 事件循环去执行其他就绪的协程
5. 当 IO 操作完成，对应的协程被放回就绪队列
6. 事件循环继续取下一个协程执行
7. 直到所有协程完成

**关键理解**：整个过程只在**一个线程**中运行。依赖操作系统提供的 IO 多路复用机制（epoll/kqueue/IOCP）来同时监听多个 IO 事件。

## 1.3 核心概念区分

| 概念 | 定义 | 关键特征 |
|------|------|---------|
| 协程（Coroutine） | `async def` 定义的函数 | 可以暂停和恢复的函数 |
| Task | 被事件循环调度的协程 | 创建后立即开始执行，可被取消 |
| Future | 一个未来才完成的值 | 底层抽象，Task 的基类 |
| await | 等待一个可等待对象完成 | 暂停当前协程但不阻塞线程 |
| gather | 并发运行多个协程，等全部完成 | 返回所有结果的列表 |
| create_task | 创建后台任务，不等它完成 | 用于"发射后不管"的场景 |

## 1.4 为什么要用异步而不是多线程

AI 应用是典型的 **IO 密集型**场景——等待 LLM API 响应、查询向量数据库、读 Redis 缓存。这些操作 90% 的时间花在等 IO，CPU 空转。

- **GIL（全局解释器锁）**：Python 的多线程在同一时刻只有一个线程能执行 Python 代码。IO 密集型场景下 GIL 不是问题，但线程切换有开销。
- **协程开销极低**：协程切换在用户态完成，不涉及内核调度，切换成本在微秒级。单线程可以维持上万连接。
- **线程开销大**：每个线程占用 8MB 栈空间，创建和销毁成本高。大量线程下切换开销显著。

## 1.5 常见错误模式

- 在协程中调用同步阻塞函数（如 `requests.get()`）会阻塞整个事件循环
- 忘记 `await`——协程不 await 就不会执行，静默跳过
- 并发太大没有限流——同时发 1000 个请求可能触发 API 限流或被封禁
- 用 `asyncio.run()` 在一个已有事件循环的环境中（Jupyter、FastAPI）

## 1.6 面试要点速记

- **并发（Concurrency）≠ 并行（Parallelism）**：并发是交替执行（协程），并行是同时执行（多进程/多机）
- **协程是协作式的**：只在 `await` 点主动让出控制权，不会像线程那样被操作系统抢占
- **Python 异步生态**：`httpx`（异步 HTTP）、`asyncpg`（异步 PostgreSQL）、`redis.asyncio`（异步 Redis）

# 第二章：FastAPI 服务构建

## 2.1 FastAPI 为什么适合 AI 服务

- **原生异步**：所有请求处理函数都支持 `async def`，是 Python 异步框架中性能最高的
- **自动文档**：基于 OpenAPI，自动生成 Swagger UI 交互式文档
- **Pydantic 集成**：请求/响应验证、序列化、类型安全，AI 应用数据模型复杂，Pydantic 大幅降低 Bug
- **SSE 原生支持**：`StreamingResponse` 直接实现 LLM 流式输出
- **依赖注入**：可复用的认证、数据库连接、配置管理

## 2.2 请求处理流程

```
用户请求 → Uvicorn（ASGI 服务器，处理网络连接）
         → FastAPI 路由匹配
         → 中间件（日志、限流、认证、CORS）
         → 依赖注入（获取 DB 连接、验证 Token、加载配置）
         → 请求处理函数（业务逻辑，调用 LLM / 向量库）
         → Pydantic 响应序列化
         → 返回 JSON / SSE 流
```

## 2.3 Pydantic 的 AI 应用价值

AI 应用的数据模型通常很复杂——OpenAI 的响应有十几层嵌套、Tool Call 参数动态变化、不同模型返回不同格式。Pydantic 提供：

- **类型安全**：编译时（IDE 提示）+ 运行时双重验证
- **数据转换**：自动将 JSON 字符串/日期字符串转换为 Python 对象
- **嵌套模型**：定义了 ToolCall → Function → Parameters 三层结构，自动校验
- **field_validator**：自定义校验逻辑，如检查 temperature 是否在 [0, 2] 范围内

## 2.4 生产环境注意事项

- **Uvicorn + Gunicorn**：多 Worker 模式，Gunicorn 管理进程生命周期，每个 Worker 运行 Uvicorn
- **超时配置**：LLM 调用可能耗时 30 秒，Keep-Alive 超时和请求超时要设够
- **健康检查**：提供 `/health`（检查服务是否启动）和 `/ready`（检查依赖服务是否可用）
- **优雅关闭**：收到 SIGTERM 时停止接收新请求，等当前请求处理完再退出

# 第三章：向量数据库与相似度检索

## 3.1 Embedding 的本质

Embedding（向量嵌入）是将文本映射为高维空间中的点。它的核心假设是：**语义相似的文本在向量空间中距离更近**。

- 这种映射是由神经网络模型学习得到的
- 维度通常是 256/512/768/1024/1536/3072
- 同一模型生成的向量才能比较（不同模型的向量空间不同）
- 向量本身没有可解释性——不能说"第 3 维代表情感"

## 3.2 相似度度量深度对比

**余弦相似度（最推荐）**：
- 衡量两个向量方向的相似度，忽略长度
- 取值范围 [-1, 1]，1 表示完全相同，0 表示正交（无关），-1 表示完全相反
- 适合文本嵌入：两句意思相近的话，向量方向应该相近，即使一个是长句一个是短句
- 大多数 Embedding 模型输出 L2 归一化后的向量，此时余弦相似度 = 点积

**欧几里得距离**：
- 衡量空间中两点的直线距离
- 值越小越相似
- 对向量长度敏感——高频词的向量通常更长，欧氏距离容易偏向高频词
- 向量 L2 归一化后，欧氏距离和余弦相似度单调相关

**点积（内积）**：
- 衡量方向和长度的综合
- 向量越长（模越大），点积越大
- 适合用对比学习训练的模型，不适合未归一化的模型

## 3.3 pgvector 深度使用

pgvector 是 PostgreSQL 的向量扩展，让一张表里同时存储结构化字段（日期、分类、作者）和向量，支持混合查询——一条 SQL 同时做向量检索和结构化过滤。

**索引选择**：

| 索引 | 原理 | 特点 |
|------|------|------|
| **IVFFlat** | 将向量空间分区（lists），查询时只搜最近的分区 | 速度快 10-100 倍，召回不是 100%，需要定期重建 |
| **HNSW** | 多层图结构，每层图上的最近邻搜索 | 速度更快精度更高，建索引慢，内存占用大 |

**lists 参数**：分区数，建议值 = 总行数的平方根。太小精度高但慢，太大速度快但精度下降。

## 3.4 向量数据库选型决策树

```
项目阶段？
  原型/POC → Chroma（最轻量，Python 一行代码启动）
  生产环境 →
    已有 PostgreSQL？→ 用 pgvector（零运维增量）
    自建新服务 →
      数据量 < 1 亿条 → Qdrant / Weaviate
      数据量 > 1 亿条 → Milvus（分布式 + GPU 加速）
      不想管运维 → Pinecone（全托管，付费）
```

## 3.5 混合检索（Hybrid Search）

单一向量检索的问题：对专有名词、缩写、精确 ID 的搜索不如关键词匹配。

**混合检索**：同时进行向量检索和 BM25 关键词检索，将两个结果集融合排序。

**融合方式**：
- **RRF（倒数排名融合）**：按每个文档在两组结果中的排名倒数加权求和
- **权重融合**：语义分 × 0.7 + 关键词分 × 0.3

# 第四章：大语言模型核心概念

## 4.1 Prompt Engineering 的层次

- **初级**：写一句指令让模型做事
- **中级**：设计角色、规则、输出格式、Few-shot 示例
- **高级**：理解模型的认知偏见，通过 Prompt 引导推理路径（CoT、ReAct、Tree of Thoughts）

## 4.2 System Prompt 的设计原则

System Prompt 的优先级高于 User Message，模型更倾向于遵守。设计要点：

1. **角色定义**："你是一个..."——激活模型在特定领域的知识
2. **行为约束**："不要...""必须..."——设定安全边界
3. **输出格式**："返回 JSON""用 Markdown 格式"——降低解析难度
4. **知识范围**："只使用提供的参考资料""如果不知道就说不知道"——控制幻觉

## 4.3 Few-shot 为什么有效

Few-shot 不是给模型"新知识"，而是给模型"输出模式"。模型通过示例理解你想要的结构、语气、格式。核心是**示例质量**而非数量，通常 2-5 个精挑细选的示例比 20 个随机示例效果好。

## 4.4 Chain-of-Thought（思维链）

让模型在输出最终答案之前进行逐步推理。可以显著提高复杂推理任务的准确性。

**原理**：LLM 是自回归模型——先输出的 token 影响后面的 token。当模型在开头产出推理步骤时，这些推理步骤作为上下文参与后续的生成，帮助模型得出更准确的结论。

**变体**：
- **Zero-shot CoT**：不加示例，直接加一句 "Let's think step by step"
- **Few-shot CoT**：给出带推理步骤的示例

## 4.5 Hallucination（幻觉）的成因与缓解

**成因**：
- 模型本质是统计语言模型，不是知识库——它预测最可能的下一个 token，不"知道"真假
- 训练数据截止日期之后的事件模型完全不知道
- 模型倾向于给出确定答案，即使不确定也会编

**缓解手段**：
- **RAG**：以检索到的文档为依据，加入"只使用参考资料"约束
- **让模型输出置信度**：Forcing the model to express uncertainty
- **降低温度（temperature）**：温度越低输出越确定（但也越重复单调）
- **事后验证**：用另一个模型或规则检查答案的事实性

## 4.6 Temperature 与 Top-p 的含义

- **Temperature**：控制输出的随机性。0 = 确定（每次一样），1 = 正常随机，2 = 极度随机
  - 数学事实问答用低温（0-0.3），创意写作用高温（0.7-1.0）
- **Top-p（Nucleus Sampling）**：限制候选 token 集合，只从累积概率达到 p 的前 N 个 token 中采样
  - p=0.1 只考虑最可能的 10% token，输出非常保守

# 第五章：RAG 检索增强生成

## 5.1 RAG 要解决的核心问题

LLM 训练数据有截止日期，且不对企业内部知识了解。RAG 让 LLM 能基于**实时、私有、可控**的知识生成答案。

## 5.2 分块策略的深层考量

分块（Chunking）是 RAG 中最被低估但影响巨大的环节。

**分块大小的 Trade-off**：
- 小块：检索精度高（找到的就是最相关的那几句），但上下文不足（断章取义）
- 大块：上下文完整，但包含太多无关信息，容易让 LLM 分心

**分块重叠的必要性**：自然语言不是按固定长度组织的——一个完整的语义单元可能在切分边界被截断。重叠让相邻块共享边界内容，减少信息丢失。

**递归分割的优势**：先从最大的语义边界（段落）尝试，不满足尺寸要求再降级到句子，最后才是固定字符切分。这样最大程度保持了语义完整性。

## 5.3 RAG 进阶策略详解

**HyDE（假设文档嵌入）**：
- 问题：用户提问太口语化、太短，直接拿去检索命中率低
- 做法：先让 LLM 根据问题生成一个假设的答案文档，用这个假设文档做向量检索
- 原理：假设文档和真实文档在语言风格和结构上更接近，比短问题更容易匹配
- 代价：多调用一次 LLM

**Small-to-Big**：
- 检索阶段用小 chunk（如 256 字符）确保精度
- 生成阶段用包含该 chunk 的大上下文（如 1024 字符）确保完整性
- 实现：建立小块到大块的映射关系

**Multi-Query / Query Expansion**：
- 一个问题生成 3-5 个不同表述的变体
- 每个变体独立检索，合并去重结果
- 提高召回率，降低措辞偏差的影响

**Re-ranking**：
- 初检索召回 20+ 条候选（高召回）
- 用更精准的模型（Cross-Encoder）重新排序
- 取 Top 3-5 条送入 LLM（高精度）
- 典型的两次检索架构：粗排 + 精排

## 5.4 检索失败的五种模式

| 失败模式 | 表现 | 原因 | 解决 |
|---------|------|------|------|
| 完全遗漏 | 找不到任何相关文档 | 知识库没有、Embedding 模型不匹配 | 扩展知识库、换模型 |
| 部分遗漏 | 找到部分，漏了关键文档 | 查询不完整、chunk 太碎 | 查询重写、Multi-Query |
| 噪音过多 | 找到很多但大部分无关 | 知识库混乱、相似度阈值太低 | 阈值过滤、Re-rank |
| 排名错误 | 相关文档在，但排到了后面 | 相似度度量有偏向 | Re-rank、混合检索 |
| 信息过时 | 文档版本陈旧 | 缺乏更新机制 | 版本管理、增量索引 |

# 第六章：Function Calling / Tool Use

## 6.1 工作原理

Function Calling 不是让 LLM 执行函数——LLM 只输出一个结构化的 JSON 描述"我想调哪个函数、传什么参数"。你的代码接收这个 JSON 后实际执行函数，把结果返回 LLM 继续生成。

**关键理解**：模型在训练时学会了如何从自然语言中提取结构化参数。它分析用户的自然语言请求，识别意图，填充函数需要的参数。

## 6.2 设计原则

**函数描述的质量决定调用准确率**：
- 函数名要语义清晰（`search_documents` 而不是 `func1`）
- 描述要说明什么时候该调用这个函数
- 参数描述要足够具体，最好给示例

**参数设计**：
- 尽量用枚举（enum）而非自由文本——减少 LLM 编造参数值
- 必填参数设 `required`
- 参数类型越精确越好

## 6.3 调用模式

- **单次调用**：LLM 只返回一个 tool_call，执行后返回结果，LLM 生成回答
- **多轮调用**：LLM 调用工具 → 得到结果 → 再次调用另一个工具 → 直到完成任务
- **并行调用**：LLM 一次返回多个 tool_call，适合相互独立的操作（同时查两个城市的天气）
- **强制调用**：设置 `tool_choice="required"` 强制 LLM 必须调用工具

## 6.4 错误处理模式

当工具执行失败时，不应该直接向用户报错，而应该把错误信息返回给 LLM，让 LLM 决定如何调整。例如：
- 参数错误 → LLM 可能修正参数重新调用
- 服务不可用 → LLM 可能换用另一个工具
- 超时 → LLM 可能缩小查询范围

# 第七章：Agent 架构

## 7.1 Agent 与普通 LLM 调用的本质区别

普通 LLM 调用是**单次推理**，Agent 是**多步决策循环**。

| 维度 | 普通 LLM | Agent |
|------|---------|-------|
| 执行模式 | 输入→输出，一次性 | 观察→思考→行动，循环 |
| 外部能力 | 无 | 调用工具、数据库、API |
| 记忆 | 只有当前对话 | 短期+长期记忆 |
| 规划 | 无 | 任务分解、子目标 |
| 纠错 | 出错就出错 | 观察行动结果并调整 |

## 7.2 Agent 的核心能力

**规划（Planning）**：将复杂任务分解为可执行的子任务，决定执行顺序。

**工具使用（Tool Use）**：知道什么时候用哪个工具，怎么用。

**记忆（Memory）**：
- 短期记忆：当前对话上下文
- 长期记忆：跨会话的持久化信息（用户偏好、历史决策）
- 工作记忆：当前任务的中间状态

**反思（Reflection）**：评估自己的输出质量，发现错误并纠正。

## 7.3 常见架构模式

**ReAct**：最基础的 Agent 模式，交替进行推理和行动。适合明确的任务。

**Plan-and-Execute**：先整体规划再逐步执行，适合复杂多步任务。

**Reflexion**：在 ReAct 基础上增加反思环节——执行后评估结果，失败时分析原因并修改策略。

**AutoGPT / BabyAGI 风格**：给一个终极目标，Agent 自主规划和执行，适合开放式的长期任务。

## 7.4 Agent 的可靠性挑战

Agent 最大的问题是**可靠性**——多步决策每一步都可能出错，错误会传导和放大。

- **幻觉调用**：Agent 编造了不存在的工具名或参数
- **任务漂移**：Agent 偏离原始目标，执行了不相关的操作
- **无限循环**：Agent 反复调用同一工具没有进展
- **过度工具化**：简单闲聊也要调工具，用户体验差

**可靠性提升手段**：严格限定 max_steps、对工具进行参数校验、用更好的模型做规划、关键操作需确认。

# 第八章：LangChain 与框架生态

## 8.1 核心抽象理解

**Chain**：多个组件的顺序组合。最简单的 Chain = Prompt + LLM + Output Parser。

**LCEL（LangChain 表达式语言）**：用 `|` 管道符串联组件，表达数据从 A 流到 B 到 C。

**Agent**：用 LLM 驱动决策的 Chain，动态选择下一步操作而非固定流程。

**Tool**：LangChain 将任意可调用对象封装为 Tool，Agent 可以调用。关键装饰器是 `@tool`。

**Memory**：对话历史管理。LangChain 提供了多种 Memory 策略（缓冲窗口、摘要记忆、混合）。

## 8.2 关键设计模式

**Stuff**：把所有检索到的文档全部塞进 prompt（简单但受限于上下文窗口）

**Map-Reduce**：每篇文档单独总结，再汇总所有总结（适合多文档）

**Refine**：逐篇文档迭代优化答案（质量高但慢）

**Map-Rerank**：每篇文档打分，用最高分的文档生成答案

## 8.3 优缺点深度分析

**LangChain 真正做对了什么**：
- 降低了 RAG 和 Agent 的入门门槛
- LangSmith 的可视化调试对理解复杂调用链很有价值
- 推动了 LLM 应用开发模式的标准化

**LangChain 的真实问题**：
- 过度抽象：为了通用性牺牲了可理解性。一个简单的操作被封在 5 层类继承后面
- 隐性行为：很多操作有副作用（比如自动管理对话历史），文档没说清楚
- 版本碎片化：0.1 → 0.2 再到现在的版本，API 一直变，旧教程都失效

## 8.4 LangChain vs LlamaIndex

**LangChain** 是通用 LLM 应用框架——它关心的是"怎么把 LLM、工具、记忆组装成应用"。适合构建 Agent、Workflow、多步骤对话。

**LlamaIndex** 是专注数据索引和检索的框架——它关心的是"怎么把数据组织好让 LLM 能用"。适合构建知识库、文档问答、数据 Agent。

它们是互补关系。很多人用 LlamaIndex 做数据层，用 LangChain 做应用编排层。

# 第九章：MCP（Model Context Protocol）

## 9.1 本质理解

MCP 是一个**标准化的 API 协议**，让 AI 应用能通过统一接口调用各种工具和数据源。就像 USB 让电脑通过同一种接口连接不同外设。

**没有 MCP 之前**：每个 AI 应用要单独写代码对接每个工具。10 个应用 × 5 个工具 = 50 个对接实现。

**有了 MCP 之后**：工具一次实现 MCP Server 接口，所有 MCP 兼容的 AI 应用都能直接调用。10 个应用 + 5 个工具 = 15 个实现。

## 9.2 协议架构

```
┌──────────────┐                ┌──────────────┐
│ MCP Client    │  JSON-RPC 2.0  │ MCP Server    │
│ (AI 应用)     │ ←───────────→ │ (工具/数据源)  │
└──────────────┘   stdio/HTTP   └──────────────┘
```

## 9.3 三个核心原语

**Tools**：AI 可以调用的函数。类比 REST API 的 POST。模型发起调用，服务端执行。

**Resources**：AI 可以读取的数据。类比 REST API 的 GET。服务端暴露数据，客户端读取。

**Prompts**：预定义的提示模板。服务端提供模板，客户端使用。

## 9.4 当前状态（2026年）

- Anthropic 主推，已发布正式规范
- Claude 系列产品（Claude Desktop、API）原生支持 MCP Client
- 社区已经贡献了大量 MCP Server（数据库、文件系统、各类 SaaS）
- 其他大模型厂商的态度还需观察——如果只有 Anthropic 生态支持，协议的网络效应有限

# 第十章：部署与运维

## 10.1 AI 服务的特殊性

- **长连接占比高**：SSE 流式输出占用比普通 HTTP 长得多的连接时间
- **负载模式不可预测**：用户活跃时间、突发话题（热点事件导致问答激增）
- **第三方依赖多**：依赖 LLM API、Embedding API、向量库、Redis、数据库
- **成本直接与流量相关**：不是固定成本，每个请求产生实际 API 费用

## 10.2 K8s 自动扩缩的挑战

CPU 指标对 AI 服务不太适用——等待 LLM 响应时 CPU 几乎空闲，但 Pod 的连接数已经满了。需要用自定义指标（活跃连接数、请求排队长度、P95 延迟）驱动 HPA。

## 10.3 LLM 缓存的分层设计

**两层缓存架构**：
- L1 精确缓存：Redis 存 SHA256(prompt)→response，命中率 5-10%，命中时 1ms 返回
- L2 语义缓存：向量库存 prompt_embedding→response，相似度 > 0.95 命中，命中率 20-40%
- L1 命中无需调 Embedding，L2 命中需要调 Embedding（但比调 LLM 便宜 500 倍）

## 10.4 可观测性

AI 应用需要比传统应用更细致的监控：
- **Token 用量追踪**：按用户/按模型/按时间的 Token 消耗和费用
- **LLM 调用延迟**：P50/P95/P99 延迟，按模型分
- **缓存命中率**：L1 和 L2 各自的命中率
- **检索质量监控**：检索结果的平均相似度分数分布
- **错误分类**：限流错误、超时错误、内容拒答、参数错误

## 10.5 限流策略

**为什么要限流**：
- LLM API 有速率限制
- 防止单个用户消耗过多成本
- 防止滥用和攻击

**限流层级**：
- 全局限流：整个服务的 LLM API 调用上限
- 用户限流：每个用户每分钟/每小时/每天的限制
- IP 限流：防止恶意刷量

# 第十一章：安全

## 11.1 Prompt Injection 深层理解

Prompt Injection 本质是**数据与指令的混淆**——LLM 无法区分哪些是指令（System Prompt 设定），哪些是数据（用户输入、检索到的文档）。

**为什么无法彻底解决**：LLM 的核心功能就是接收自然语言并执行其中的意图。阻止它执行某些意图与它的根本设计相矛盾。防护是猫鼠游戏。

**纵深防御**：不靠单一手段，而是多层配合——输入过滤识别可疑模式、架构层隔离指令和数据、输出审核拦截违规内容。

## 11.2 API Key 安全

- 永远不在代码中硬编码
- 使用环境变量或密钥管理服务（Vault、AWS Secrets Manager）
- 生产环境 API Key 按最小权限原则授予
- 轮换机制——定期更换 Key

## 11.3 数据安全

- 用户输入可能包含敏感信息——日志中脱敏处理
- 传输 LLM API 的数据经 HTTPS 加密
- 考虑数据驻留要求（某些数据不能离开特定地区）
- 向量数据库中存储的文档可能包含商业秘密——控制访问权限

# 第十二章：选型与 Trade-off

## 12.1 何时用 RAG vs 微调

| 场景 | 选 RAG | 选微调 |
|------|--------|--------|
| 知识动态变化 | ✅ 更新文档即可 | ❌ 需要重新训练 |
| 需要引用来源 | ✅ 直接追溯 | ❌ 融入了模型权重 |
| 预算有限 | ✅ 只需 Embedding | ❌ 训练成本高 |
| 领域极其专业 | ⚠️ 检索可能不准 | ✅ 模型内化领域知识 |
| 需要特定风格 | ❌ 只能靠 Prompt | ✅ 可以学到风格 |
| 数据量巨大 | ✅ 向量库可扩展 | ❌ 训练时间和成本 |

## 12.2 模型选择决策

- 简单分类/提取：GPT-3.5 / Claude Haiku / 小开源模型（Qwen 等）
- 日常对话/问答：GPT-4 Turbo / Claude Sonnet
- 复杂推理/Agent：GPT-4 / Claude Opus
- 长文档处理（> 32K tokens）：Claude（原生 200K）
- 多语言特别是中文：Claude ≥ GPT-4 >> 开源模型（英文好但中文弱）
- 私有化部署：Llama 4 / Qwen 3 + vLLM 推理框架

## 12.3 自建 vs API vs 全托管

- **自建**：部署开源模型（Llama/Qwen），数据不出公司，无 API 调用费，但需要 GPU 运维
- **API 调用**：用 OpenAI/Anthropic 的 API，零运维，按量付费，但有网络延迟和数据出境的担忧
- **全托管平台**：Azure OpenAI / AWS Bedrock，在企业云环境内调用，兼顾便利和数据安全
