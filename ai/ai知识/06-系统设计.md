# 系统设计 — AI 应用面试高频题

## 系统设计答题框架（每次面试都用）

```
1. 需求澄清（2 分钟）
   - 功能需求：用户能做什么？
   - 非功能需求：QPS、延迟、可用性、数据量？
   - 约束条件：预算、技术栈限制？

2. 数据流设计（3 分钟）
   - 数据从哪里来、经过哪些组件、到哪里去
   - 画出框图和箭头

3. 组件拆分（5 分钟）
   - 每个组件的职责和技术选型
   - 为什么选这个方案而不是别的

4. 接口设计（3 分钟）
   - REST API / WebSocket / SSE
   - 请求和响应的数据结构

5. 深度讨论（5-10 分钟）
   - 边界情况、扩展性、故障处理
   - 面试官会深挖的一两个点
```

## 题目一：设计一个 RAG 文档问答系统

### 题目
设计一个企业内部文档问答系统。用户上传 PDF/Word/Markdown 文档，然后可以对文档内容进行自然语言提问。

### 1. 需求澄清

**功能需求：**
- 用户上传文档（PDF、Word、Markdown、TXT）
- 系统解析文档内容并建立索引
- 用户通过自然语言提问
- 系统返回带引用来源的答案
- 支持多文档查询

**非功能需求：**
- 上传后 30 秒内可搜
- 问答响应 < 5 秒
- 支持 1000 个并发用户
- 文档总大小可达 10GB

### 2. 数据流设计

```
上传流程：
用户上传文件 → API 接收 → 文件存储(S3/MinIO)
                         → 消息队列 → 文档解析 Worker
                                     → 分块(Chunking)
                                     → Embedding
                                     → 写入向量数据库

查询流程：
用户提问 → API → Embedding 模型 → 向量搜索 Top-10
                                 → Re-rank Top-3
                                 → 构建 RAG Prompt
                                 → LLM 生成
                                 → 返回(答案 + 来源引用)
```

### 3. 组件拆分与技术选型

| 组件 | 技术选型 | 理由 |
|------|---------|------|
| API 层 | FastAPI + Uvicorn | 高性能异步、原生 SSE 支持 |
| 文件存储 | MinIO (S3 兼容) | 自建成本低 |
| 消息队列 | Redis Streams / RabbitMQ | 解耦上传和处理 |
| 文档解析 | Unstructured / PyMuPDF | 支持多种格式 |
| 向量数据库 | PostgreSQL + pgvector | 已有 PG 基础设施，减少运维 |
| Embedding | OpenAI text-embedding-3 | 性价比高 |
| LLM | GPT-4 Turbo | 质量最高 |
| 重排序 | Cohere Rerank / BGE-Reranker | 提高检索精度 |
| 缓存 | Redis | 缓存热门问题答案 |

### 4. 接口设计

```python
# 上传文档
POST /api/v1/documents/upload
Request: multipart/form-data { file, collection_name }
Response: { "document_id": "doc_123", "status": "processing" }

# 查询上传状态
GET /api/v1/documents/{document_id}/status
Response: { "status": "indexed", "chunks": 45, "indexed_at": "..." }

# 问答
POST /api/v1/qa
Request: { "question": "...", "collection_name": "...", "top_k": 5 }
Response: {
    "answer": "...",
    "sources": [
        {"document": "manual.pdf", "page": 3, "excerpt": "...", "score": 0.92}
    ],
    "tokens_used": 1234
}

# 流式问答
POST /api/v1/qa/stream
Request: { "question": "...", "collection_name": "..." }
Response: SSE stream
```

### 5. 深度讨论

**Q: 如何处理文档更新？**
```
方案：版本化索引
- 每个文档有 version_id
- 更新时建新索引，完成后原子切换
- 旧版本保留 24 小时便于回滚
```

**Q: 如何提高检索精度？**
```
组合策略：
1. 混合检索：向量检索 + BM25 关键词检索，融合排序
2. Re-ranking：粗召回 20 条 → 精排模型 → Top 3
3. 元数据过滤：按部门、日期、文档类型预过滤
4. 查询重写：LLM 扩展用户问题（Multi-Query）
```

**Q: 如何处理超长文档（ > 10 万 token）？**
```
1. 递归分块：段落 → 句子 → 固定大小
2. 层级索引：摘要索引 + 详细索引两级
3. Small-to-Big：检索用小 chunk(256)，生成用大 chunk(1024)
```

**Q: 如何控制成本？**
```
1. 精确缓存：相同/相似问题不重复调用 LLM
2. 问题预检：简单问题走 FAQ 匹配，不调 LLM
3. Embedding 缓存：文档内容不变不重新向量化
4. 降级策略：高峰期用 GPT-3.5 替代 GPT-4
```

## 题目二：设计一个智能客服机器人

### 题目
设计一个电商智能客服，能处理商品咨询、订单查询、退换货等场景，复杂问题转人工。

### 数据流
```
用户消息 → 意图识别(LLM) → ┬→ FAQ 匹配(简单问题)
                           ├→ 知识库检索 RAG(商品/政策)
                           ├→ API 查询(订单/物流)
                           └→ 转人工(复杂/投诉)
                         → 答案生成(LLM + 约束)
                         → 安全审核 → 返回
```

### 关键设计

**意图识别 Prompt：**
```
你是电商客服意图分类器。分析用户消息，输出 JSON：
{
    "intent": "product_query | order_status | refund | complaint | chitchat",
    "confidence": 0.95,
    "entities": {"order_id": "12345", "product": "手机壳"}
}
```

**多轮对话状态管理：**
```python
class DialogState:
    def __init__(self):
        self.current_intent: str | None = None
        self.collected_info: dict = {}
        self.turn_count: int = 0
        self.required_fields: list[str] = []

    def is_complete(self) -> bool:
        """所有必填信息是否收集完毕"""
        return all(k in self.collected_info for k in self.required_fields)

    def next_question(self) -> str:
        """下一个要问的信息"""
        for field in self.required_fields:
            if field not in self.collected_info:
                return f"请提供您的{field}"
        return ""
```

**人工转接策略：**
```
触发条件：
- 用户连续表达不满 2 次
- 意图置信度 < 0.6
- 检测到投诉关键词
- 用户明确要求转人工

转接流程：
1. 生成会话摘要
2. 保存到 Redis（5 分钟 TTL）
3. 回复用户等待时间
4. 通知人工客服队列
```

## 题目三：设计一个 AI 代码审查工具

### 题目
设计一个自动代码审查工具，集成 GitHub，对 PR 进行 AI 审查并自动评论。

### 架构
```
GitHub Webhook → API 接收 PR 事件
              → 获取 diff
              → 分块（按文件 / 按函数）
              → 并发 AI 审查
                 ├→ 安全检查（密钥泄露、SQL 注入）
                 ├→ 代码规范检查
                 ├→ 性能问题检查
                 └→ 最佳实践建议
              → 汇总结果
              → 通过 GitHub API 发 PR 评论
```

### AI 审查 Prompt
```
你是高级代码审查员。审查以下代码变更，重点关注：
1. 安全漏洞（XSS、SQL 注入、密钥泄露）
2. 性能问题（N+1 查询、未释放资源）
3. 边界情况（空值处理、异常处理）
4. 代码可读性

仅对确实存在问题的地方评论，不要在没问题的代码上充数。

输出 JSON 格式：
[{
    "file": "src/user.py",
    "line": 42,
    "severity": "critical | warning | suggestion",
    "message": "...",
    "suggestion": "...（可直接 commit 的代码建议）"
}]
```

### 扩展讨论
- **大 PR 处理**：超过 50 个文件，跳过 trivial 变更（格式、import），只审逻辑变更
- **误报处理**：允许团队配置自定义规则、忽略特定模式
- **隐私**：代码不离开私有部署环境

## 面试实战技巧

### 面试中要主动展示的点
1. **数据流思考习惯**：每道题先画数据流，再谈技术
2. **Trade-off 意识**：没有银弹，每个选择都有代价
3. **成本意识**：AI 应用的钱 = API 调用费，要在设计中控制
4. **故障处理**：LLM 挂了怎么办？外部 API 超时怎么办？

### 常见追问
| 问题 | 答辩思路 |
|------|---------|
| "LLM 返回错误怎么办？" | 重试 + 兜底答案 + 降级模型 |
| "怎么测这个系统？" | 单元测试 + Eval 数据集 + 人工评估 |
| "怎么保证数据安全？" | 加密存储 + 脱敏 + 访问控制 + 审计日志 |
| "怎么处理高并发？" | 缓存 + 限流 + 异步队列 + 水平扩展 |
| "怎么做 A/B 测试？" | 按用户哈希分流，对比答案质量和延迟 |
